TestCase

A TestCase is an XML file. The XML structure is a further development of the new German BSI XML structure. It will be structured in the so called "ActionSteps": PreCondition, TestSteps, and PostCondition.

ActionSteps can encapsulate simple APDU commands, or call more complex java script functions that may use variables that have been defined in them.

A PreCondition may be a state of the card or simulator that can be achieved by performing some operations before the TestSteps are performed.

In TestSteps, the actual purpose of the TestCase will be implemented.

A TestStep may cover one major step of a TestCase, for example "Perform BAC", "Read Data Group XY", "Select Application Z", and so on.

Sometimes the TestSpecification requires a finer granularity, for example if an erroneous APDU is sent, and the according response APDU shall be checked. Examples are "MSE: Set AT", "General Authenticate", "PSO: Verify Certificate", etc.

A PostCondition may be a state of the card or simulator that should be achieved after the TestSteps are performed. To ensure this condition, several commands can be sent and their responses be checked. If the results comply to the ones that this certain state of the chip is supposed to send back, the PostCondition is fulfilled.

PreCondition, TestSteps, and PostCondition will be organized in the following tags:

  • Command gives the "ActionStep" a title; or APDUCommand may specify an APDU to send in addition. Variables that might be used here can be set in the "TechnicalCommand" tag.
  • Description is a textual description of what happens in this step.
  • TechnicalCommand may
    • set variables in the Java Script scope,
    • call Java Script functions, and
    • send APDU (declaration as ByteString and using the send command function).
  • ExpectedResult will evaluate the response data stored in the variable "response". Status words and response data can be checked using assertions.
For further and specific information also have a look at the XML structure for TestCases.
