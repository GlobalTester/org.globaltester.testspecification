XML structure for TestCases

A TestCase is structured in these XML tags:

  • id: ID of the test case.
  • Title: The Title of the TestCase. It should correspond to the title of the test case in its according test specification.
  • Version: The version number the test case has according to its TestSpecification.
  • Purpose: A short description of the purpose of the test case.
  • Profile: A TestSpecification often defines profiles. Each test case has some of them assigned to. They will be listed in this tag.
  • Reference (optional): References can be listed in this tag, for example the technical guideline the test specification is based on.
  • PreCondition (optional)
  • TestStep (several possible)
  • PostConditions (optional)
  • MetaData: This tag contains meta information in form of key-value pairs, for example if a test case is destructive.
  • Comment: If a test case was deleted in a later version of the test specification, this will be explained in this tag.

PreCondition, TestSteps and PostConditions are organized in the following tags:

  • Command: A tag that gives the test step a title, i.e. "Reset Card", encapsulated in a <text> tag.
    An APDUCommand is a special command that additionally specifies an APDU that will be sent in an <apdu> tag
  • Description (optional): A short description of the purpose of the test step.
  • TechnicalCommand (optional): In this tag the concrete implementation of the textual description takes place. There can be
    • Variables set within the Java Script scope,
    • functions called within the Java Script scope, or
    • APDUs declared as ByteString and sent.
    Please note: Variables that may be used in the APDUCommand are set in the TechnicalCommand. Thus, the TechnicalCommand will be evaluated first in the Java Script scope, and then the APDUCommand will be executed.
  • TestDataReference (optional): In this tag reference to some predefined test data, such as certificates, can be made.
  • ExpectedResult: In this tag the result APDU can be verified, i.e. its status words and response data (if any) can be checked. It is structured in two possible tags as well:
    • Text: Gives a short textual description of the expected result, i.e. "SW 9000" if the status word 9000 is expected.
    • TechnicalResult: Status words and response data can be checked in assertions.
